## Open Items — Canonical Pipeline

Current baseline: ~18 API calls per scorecard run; ~10-20 calls per matrix run when low-confidence recovery is active.

Priority scale:
- **P0** — reliability blocker / correctness bug / Tier 0 prerequisite before any quality revalidation.
- **P1** — strong quality lift, or core product surface that unlocks user value (auth, billing, run library).
- **P2** — strategic lift; meaningful but not blocking.
- **P3** — deep/expensive optional modes, refactors, or long-term architectural shifts.

### Current Execution Order

**Tier 0 (P0) — Infrastructure prerequisites, land BEFORE re-validating Tier 1.** The 2026-04-20 Tier 1 integration run (debug bundle `analysis-debug-bundle-2026-04-20T14-36-55-087Z.json`) failed at stage 03a without ever reaching the decision gate. Three gaps blocked root-cause diagnosis and made every bugfix cycle expensive:

1. Logging cannot attribute aborts to a layer (browser fetch vs. chunk deadline vs. stage timeout vs. provider vs. Vercel edge) — forensics required reading code to guess.
2. Every failed run re-burns provider spend from stage 01, even when only a late stage was buggy.
3. Chunking IDs are unstable (derived from subject names) so "retry chunk X" is not a meaningful unit of work.

Tier 0 closes these three gaps. Order: extend logging first (INFRA-01/02/03, QG-06) → stage-level KV cache + resume (RES-01a/b/c) → deterministic chunking (CHUNK-01). After Tier 0 ships, re-run the 9×8 integration test to validate Tier 1 (SQ-08..SQ-14, already implemented in commit `007882f`).
